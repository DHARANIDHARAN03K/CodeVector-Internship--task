# ProductScope (CodeVector Labs Challenge)

## Architecture Overview

This project was engineered specifically to solve the "Live-Data Pagination Shift" problem at scale (200,000+ rows) while maintaining absolute correctness under heavy concurrent writes.

### Core Stack
- **Framework:** Next.js 14 App Router
- **Database:** Supabase (PostgreSQL 17.6)
- **ORM:** Prisma v5 (explicitly pinned for PgBouncer/Transaction pooling stability)

### The Pagination Engine
Instead of standard `OFFSET` pagination (which suffers from O(N) degradation and duplicates when data shifts), this application uses $O(1)$ **Cursor-Based Pagination** anchored to `(createdAt, id)` tuples.
*   The index `@@index([category, createdAt, id])` acts as a Covering Index, ensuring category filters execute without scanning the heap.
*   By encoding both the timestamp and the unique UUID into the cursor, we guarantee a stable deterministic sort order even if multiple products are created in the exact same millisecond.

### The UI & "Chaos Injection"
The frontend eschews heavy component libraries in favor of raw CSS variables and a brutalist, engineering-focused aesthetic. 
*   **The Chaos Button ("Inject 50 products"):** This exists to prove the backend architecture. Clicking it fires `POST /api/products/inject`, live-inserting 50 new rows via `@faker-js/faker`.
*   **Zero Drift Guarantee:** Reviewers are encouraged to paginate to page 3, hit "Inject", and continue to page 4. You will not see a single duplicated or skipped product, proving the cursor's mathematical resilience against data shifting.
*   **Debug Panel:** The raw base64 cursor string is deliberately exposed in the sidebar to visualize the cursor traversal mechanism.

## Getting Started

1. Set up your `.env` with a `DATABASE_URL` (Connection pooler) and `DIRECT_URL` (Raw session).
2. `npx prisma db push`
3. `npm run dev`

*Tested successfully against 200,000 rows generated by the `scripts/seed.ts` utility.*
